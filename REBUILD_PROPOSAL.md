# 現状分析と「1から作り直し」再提案

## 1. 現状の把握

### 画面構成
- `index.html` + `app.js`: 施術条件入力
- `schedule.html` + `schedule.js`: 3層（大枠/中枠/小枠）配分編集
- `timer.html` + `timer.js`: 実行タイマー
- `complete.html` + `complete.js`: 完了サマリー

### 良い点（活かすべき資産）
1. **遷移導線が明確**: 入力 → 配分 → 実行 → 完了の4画面が揃っている。
2. **データ受け渡しが統一**: `sessionStorage` で payload を繋いでいる。
3. **タイマーの実運用フローを網羅**: 開始/一時停止/次へ/リセットが実装済み。
4. **配分の警告機能がある**: 親子不一致、0分警告などの基礎検証がある。

## 2. 現状の課題（「作り直し候補」）

### A. アーキテクチャ課題
1. **各画面がグローバル state + DOM直結**
   - 表示ロジックとデータロジックが密結合。
   - 検証しづらく、仕様追加時の影響範囲が読みづらい。
2. **ストレージI/Oが散在**
   - 画面ごとにキー/フォールバックの扱いが局所化しており、整合性保証が弱い。

### B. UX課題
1. **入力/配分のガードはあるが“何を直すべきか”が弱い**
   - 警告は出るが、修正導線（ジャンプ・自動補正候補）がない。
2. **タイマー画面の進捗説明が最小限**
   - 全体残時間、現在ステップ/全ステップ、完了予測時刻などがない。

### C. 品質課題
1. **テスト基盤がない**
   - 分配ロジック・キュー生成ロジックが回帰しやすい。
2. **型/スキーマの単一定義がない**
   - 画面間 payload の互換性が実行時まで分からない。

## 3. 「1から作り直す」際の現実的な方針

### 結論
**完全スクラッチではなく、ドメインロジック中心に再設計する“段階的リビルド”を推奨**。

理由:
- 現在の画面遷移と体験骨格は有効。
- 全面作り直しは学習コストと不具合リスクが上がる。
- まずデータモデル/ロジックを切り出すだけで品質を大きく改善できる。

## 4. 再提案アーキテクチャ（最小構成）

### 4.1 レイヤ分離
- `domain/`:
  - `models.js`（Condition, Frame, Block, Unit, TimerPayload）
  - `validators.js`（入力/配分整合チェック）
  - `allocation.js`（配分・差分計算）
  - `execution-queue.js`（units > blocks > frames 優先でキュー生成）
- `infra/`:
  - `storage.js`（キー定義、保存/読込、バージョン管理）
- `ui/`:
  - 既存 `app.js`, `schedule.js`, `timer.js`, `complete.js` は UI 操作中心へ薄くする

### 4.2 データ契約
- `schemaVersion` を payload に必須化。
- 画面遷移時に必ずスキーマ検証。
- 不正時は復旧導線（入力画面へ戻す + 理由表示）。

### 4.3 テスト戦略
- ユニットテスト対象（優先順）
  1. `validators`
  2. `execution-queue`
  3. `allocation`
- E2E最小1本
  - 条件入力→配分確定→タイマー開始→完了画面

## 5. 実行計画（3フェーズ）

### Phase 1（短期: 1〜2日）
- ドメイン関数を新規作成。
- 既存画面からドメイン関数を呼ぶ形に差し替え。
- 既存UIはほぼ維持（見た目変更なし）。

### Phase 2（中期: 2〜4日）
- 配分画面の警告UX改善（修正候補、該当項目へ移動）。
- タイマー情報強化（全体残時間、進捗表示）。

### Phase 3（中期: 1〜2日）
- 完了画面の履歴要約改善。
- READMEに仕様・データ契約・テスト手順を明文化。

## 6. 次に着手する具体タスク（提案）

1. `domain/execution-queue.js` を切り出し、`timer.js` のキュー生成を移設。  
2. `infra/storage.js` を作り、`sessionStorage` 直接参照を集約。  
3. `schedule.js` の警告計算を `domain/validators.js` に移設。  
4. 最低限のテストランナー導入（Vitest想定）と3本のユニットテスト追加。  

---

この計画なら「1から作り直す」意図（構造を健全化）を満たしつつ、現状の動く価値を失わず進められます。
