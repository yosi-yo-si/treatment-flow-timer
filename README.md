# treatment-flow-timer

## 確定後フロー仕様

- `schedule.html` の「この内容で確定」押下時、現在の `TimerSchedulePayload` を `sessionStorage` の `treatmentScheduleOutput` に保存します。
- 保存後は `timer.html` へ遷移し、確定した配分をそのままタイマー実行します。
- `timer.html` では、`units` → `blocks` → `frames` の優先順で実行対象を決定します（より細かい単位を優先）。
- `timer.html` では開始・一時停止・次へ・リセットの操作が可能です。

## 次にやること（開発者向け）

1. **手動スモークテストを実施**
   - `index.html` → `schedule.html` → `timer.html` → `complete.html` を1周し、画面遷移とデータ保持を確認。
   - `bodywork_foot` など複数ケースで `units > blocks > frames` 優先が崩れないか確認。
2. **ユニットテスト基盤を追加**
   - `domain/execution-queue.js` の優先順・0分除外・秒変換をテスト化。
   - `infra/storage.js` の不正JSONフォールバックをテスト化。
3. **次フェーズの切り出し**
   - `schedule.js` の警告計算を `domain/validators.js` として分離。
   - 画面側は描画とイベント処理に限定する。

## いま迷ったら（最短）

- まずは **手動スモークテスト** を先に完了させるのが最優先です。
- その次に **`execution-queue` のテスト追加** を行うと、今後の改修の安全性が一気に上がります。

## このリファクタは「本当に必要だったか？」

結論として、**必須ではないが、中長期ではやる価値が高い**です。

- **短期（今すぐ動かす）だけが目的**なら、旧構成のままでも成立します。
- **今後も機能追加・修正を続ける**なら、`storage` と `execution-queue` の分離は回帰防止に効きます。

判断基準:

1. 今後1か月で仕様変更が複数入る見込みがある → **実施しておく価値が高い**
2. 単発運用でほぼ凍結予定 → **優先度は低い**

迷う場合は、いったん現状の分離は維持したまま、先にテストを足して効果を確認するのが安全です。
